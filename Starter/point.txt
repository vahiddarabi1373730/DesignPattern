مزایا:
۱. رعایت قانون Open/Closed Principle (OCP - اصل باز/بسته)
این قانون می‌گوید: «کلاس‌ها باید برای توسعه باز (Open) اما برای تغییر بسته (Closed) باشند.»
بدون Factory Method: اگر می‌خواستید در متد ارسال پیام، نوع ارسال را مشخص کنید، احتمالاً یک switch-case یا if-else طولانی می‌نوشتید:

  public void Send(string type, string message)
  {
      INotification notification;
      if (type == "Email")
          notification = new SendEmail();
      else if (type == "SMS")
          notification = new SendSms();
      // هر بار که کانال جدیدی (مثل Push, WhatsApp) اضافه شود، مجبورید این کد را ویرایش کنید!
  }
  با Factory Method: اگر فردا بخواهید سیستم ارسال از طریق WhatsApp را اضافه کنید، به کدهای قبلی (کلاس‌های Email و SMS و Creator اصلی) دست نمی‌زنید. فقط دو کلاس جدید می‌سازید:
  public class SendWhatsApp : INotification { ... }
    public class WhatsAppCreator : NotificationCreator { ... }
    
    سیستم قبلی بدون تغییر، کانال جدید را می‌پذیرد.
    

///////////////////////////////////////////////////////////////////////////////////////////////////////////////////////////////////////

۲. جداسازی وابستگی‌ها (Loose Coupling - جفت‌شدگی ضعیف)

کلاس اصلی شما یعنی NotificationCreator هیچ اطلاعی ندارد که قرار است SendEmail ساخته شود یا SendSms. این کلاس فقط با اینترفیس INotification کار می‌کند.

در واقع، منطق ساخت شیء (Instantiation) کاملاً از منطق استفاده از شیء (Business Logic) جدا شده است:
متد SendNotification در کلاس پدر، کارش را انجام می‌دهد بدون اینکه بداند پشت صحنه چه کلاسی ساخته شده است.

۳. کپسوله‌سازی فرآیند ساخت (Encapsulation of Object Creation)

با این الگو، تمام این پیچیدگی‌های ساختن در متد CreateNotification در کلاس‌های فرزند (EmailCreator یا SMSCreator) پنهان می‌شود و کلاینت درگیر جزئیات ساخت شیء نمی‌شود.


برای درک ساده‌ی «کپسوله‌سازی فرآیند ساخت»، بیایید به این فکر کنیم که در دنیای واقعی، ساختن بعضی اشیاء فقط با نوشتن یک کلمه new ساده تمام نمی‌شود و نیاز به مقدمات زیادی دارد.

فرض کنید کلاس SendEmail برای ساخته شدن، نیاز به اطلاعات کاربری، آدرس سرور SMTP، پورت، پروتکل امنیتی SSL و یک سرویس لاگر (Logger) دارد.

سناریوی اول: بدون Factory Method (بدون کپسوله‌سازی)
اگر از Factory Method استفاده نکنید، هر جای برنامه که بخواهید یک ایمیل بفرستید، باید تمام این جزئیات سخت‌افزاری و تنظیمات را خودتان دستی بسازید و پاس بدهید:

var smtpConfig = new SmtpSettings("smtp.gmail.com", 587, true);
var logger = new ConsoleLogger();
var authService = new AuthenticationService("user", "pass");

// تازه بعد از آماده کردن این‌ها، شیء را می‌سازید:
INotification email = new SendEmail(smtpConfig, logger, authService); 
email.SendMessage("Hello!");

مشکل: اگر فردا تصمیم بگیرید پارامترهای ساخت SendEmail را تغییر دهید (مثلاً یک پارامتر جدید اضافه کنید)، باید تمام فایل‌های پروژه‌تان که این شیء را ساخته‌اند پیدا کنید و تغییر دهید.




سناریوی دوم: با Factory Method (کپسوله‌سازی شده)
با پترن شما، تمام این کدهای کثیف و پیچیده را داخل کلاسِ سازنده‌ی مخصوص خودش (یعنی EmailCreator) پنهان (کپسوله) می‌کنیم:

public class EmailCreator : NotificationCreator
{
    public override INotification CreateNotification()
    {
        // تمام مراحل پیچیده ساخت شیء را اینجا مخفی می‌کنیم:
        var smtpConfig = new SmtpSettings("smtp.gmail.com", 587, true);
        var logger = new ConsoleLogger();
        var authService = new AuthenticationService("user", "pass");

        return new SendEmail(smtpConfig, logger, authService);
    }
}

حالا در برنامه اصلی (کلاینت)، شما اصلاً کاری ندارید که برای ساختن یک ایمیل چه چیزهایی نیاز است. به ساده‌ترین شکل ممکن فقط می‌نویسید:
NotificationCreator creator = new EmailCreator();

// فرآیند ساخت کاملاً پنهان (Encapsulated) است و شما فقط پیام را می‌فرستید:
creator.SendNotification("Hello!");

مزیت: اگر نحوه ساختن کلاس SendEmail تغییر کند، شما فقط و فقط متد CreateNotification را در کلاس EmailCreator تغییر می‌دهید و بقیه بخش‌های برنامه شما (مثل متد SendNotification یا کدهای سمت کلاینت) بدون کوچک‌ترین تغییری به کار خود ادامه می‌دهند.
